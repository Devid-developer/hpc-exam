# Optimization of a Serial, OpenMP, MPI, and Hybrid 2D Stencil

## 1. Scope and context

The program models the diffusion of energy over a two-dimensional grid using a
five-point stencil. At each iteration, the value of an internal cell `(i,j)` is
computed from its current value and its four north, south, east, and west
neighbors.

This implementation originates from the serial and parallel templates provided
for the exam and supports four execution modes:

- serial execution without the OpenMP runtime;
- shared-memory OpenMP execution;
- distributed-memory MPI execution;
- hybrid MPI+OpenMP execution.

The final benchmarks were run on the Booster partition of CINECA's Leonardo
system. Each Booster node contains one 32-core Intel Xeon Platinum 8358 Ice
Lake CPU, 512 GiB of DDR4-3200 memory, and four NVIDIA A100 GPUs, which are not
used by this CPU implementation. OpenMP was therefore limited to 32 physical
threads per node, while node identity, compiler, compilation options, and
binding were recorded. Current hardware information is available in the
[CINECA Leonardo documentation](https://docs.hpc.cineca.it/hpc/leonardo.html).

The final campaign uses up to 16 Booster nodes, with four MPI ranks and eight
OpenMP threads per rank on every node. The node count is therefore the only
resource that changes in the final strong- and weak-scaling measurements: the
per-node configuration, binding, and number of active cores remain fixed.

The coefficients are defined by the project specification:

```c
ALPHA    = 0.6
C_CENTER = 0.6
C_NEIGH  = 0.25 * (1.0 - ALPHA) = 0.1
```

The kernel consequently applies:

```text
new(i,j) = 0.6 old(i,j)
         + 0.1 [old(i-1,j) + old(i+1,j) + old(i,j-1) + old(i,j+1)]
```

The weights sum to `0.6 + 4*0.1 = 1`. Without sources and with periodic
boundaries, the operator neither creates nor destroys energy. With
non-periodic boundaries, zero-valued halos model an external heat sink.

## 2. Data layout and double buffering

The physical domain has size `xsize * ysize`. Each plane includes a one-cell
halo on every side, giving an allocated size of:

```text
(xsize + 2) * (ysize + 2)
```

The linear index is:

```c
IDX(i,j) = (size_t)j * (xsize + 2) + i;
```

Two planes, `OLD` and `NEW`, are maintained. An iteration reads exclusively
from `OLD` and writes to `NEW`; their logical roles are swapped afterward.
Double buffering avoids full-grid copies and, more importantly, removes
read-after-write dependencies within an update. Every new cell depends only on
the previous iteration, so updates are independent and suitable for both SIMD
vectorization and OpenMP parallelization.

## 3. Comparison with the initial template

### 3.1 Algebraic simplification of the kernel

The template computed `result`, `sum_i`, and `sum_j` separately and repeated
divisions and operations involving `1-alpha` inside the nested loops. The final
version evaluates the coefficients once and uses a single expression:

```c
result[i] = cc * center[i]
          + cn * (north[i] + south[i] + center[i-1] + center[i+1]);
```

For `alpha=0.6`, this transformation is algebraically equivalent, but it uses
fewer temporaries and exposes loop-invariant operations to the compiler. The
underlying principle is *loop-invariant code motion*: values that do not depend
on `i` or `j` should be computed outside the hot loop. An optimizing compiler
may perform part of this transformation automatically, but a simple expression
also improves analysis, vectorization, and maintainability.

The adopted model counts six floating-point operations per cell: four additions
and two multiplications. FMA instructions may reduce the hardware instruction
count, but they do not change the conventional algorithmic FLOP count.

### 3.2 Row pointers and reduced index arithmetic

In the template, every access expanded the `IDX` macro and repeated the
`j*fxsize` multiplication. The optimized version creates four pointers at the
beginning of each row:

```c
center = old + j*fxsize;
north  = old + (j-1)*fxsize;
south  = old + (j+1)*fxsize;
result = new + j*fxsize;
```

The inner loop then uses simple offsets. Since `i` is the innermost index,
memory accesses are contiguous. This matches C row-major layout and favors:

- spatial locality;
- hardware prefetching;
- efficient cache-line use;
- contiguous SIMD loads and stores.

### 3.3 `restrict` pointers

Plane and row pointers are qualified with `restrict`. This states that objects
read through `old` are not modified through `new` during the kernel. Without
this information, the compiler must consider possible aliasing and may need to
version the loop or skip some transformations.

The guarantee follows directly from double buffering: the two planes are
separate, non-overlapping allocations.

### 3.4 Removal of `register`

The template repeatedly used the historical `register` qualifier. Modern
compilers allocate registers from their intermediate representation and the
observed register pressure; the qualifier is not a useful optimization
strategy. It was removed to make the code clearer.

### 3.5 Index types

Expressions that produce linear offsets use `size_t`. This is the natural type
for object sizes and offsets and prevents a row-times-stride product from being
evaluated in the narrower `int` domain before conversion.

## 4. Memory allocation and initialization

### 4.1 Two separate allocations

The template used one `malloc` twice the size of a plane and placed `NEW` in
its second half. The release function freed only the initial pointer. The final
version explicitly allocates and releases two independent planes.

This makes the non-aliasing property required by `restrict` evident, simplifies
partial-failure cleanup, and lets both buffers be managed independently.

### 4.2 64-byte alignment

`posix_memalign` replaces `malloc` and aligns the beginning of each plane to 64
bytes:

```c
posix_memalign(&plane, 64, frame * sizeof(double));
```

Sixty-four bytes is the typical cache-line size of the Intel CPUs considered.
This does not guarantee that every row is aligned, because the stride includes
two halo elements, but it provides a known aligned base and avoids possible
penalties at the beginning of the allocation. The actual benefit should be
confirmed through benchmarks and compiler vectorization reports.

### 4.3 Initialization and first touch

Both planes are initialized to zero. When OpenMP is enabled, initialization is
also statically distributed among threads. On NUMA systems, the *first-touch*
policy normally places a physical page in the NUMA domain of the core that
first writes it. Matching the initialization distribution to the kernel can
therefore improve memory affinity during multicore execution.

The same code remains serial when executed with one thread. Correct pinning is
nevertheless important on the Booster. Measurements on node `lrdn1549` show
one physical socket split through Sub-NUMA Clustering into two NUMA domains:
cores `0-15` and `16-31`. This is not a socket boundary, but it is an internal
NUMA boundary. First touch and affinity should distribute pages and threads
consistently across both domains. With `OMP_PROC_BIND=close`, executions with
up to 16 threads may remain inside one domain and use only part of the
available bandwidth; using more than 16 cores brings the second domain into
play.

## 5. OpenMP parallelization

The outer row loop is parallelized with:

```c
#pragma omp parallel for schedule(static)
```

Each iteration writes a different `NEW` row and reads only from `OLD`, so there
are no races between rows. `schedule(static)` assigns deterministic iteration
blocks with low overhead. Every row has the same workload, so dynamic
scheduling would add runtime cost without improving load balance.

Directives are enclosed in `#ifdef _OPENMP`. The same source can be compiled
without OpenMP and without unknown-pragma warnings, or with `-fopenmp`.
Single-core tests of the OpenMP build set one thread, while multicore tests vary
`OMP_NUM_THREADS`.

Total energy is computed through a reduction:

```c
#pragma omp parallel for schedule(static) reduction(+:tot)
```

Each thread accumulates a private total, and the runtime combines partial sums
at the end. This avoids a race on the shared variable and does not require a
critical section for every element. Floating-point summation order can change
with the thread count, so differences in the last few bits remain possible
even when the result is numerically correct.

## 6. Boundary conditions and sources

Halos keep the inner kernel free from branches that distinguish interior and
boundary cells. With non-periodic boundaries, halos remain zero. With periodic
boundaries, after every update:

- the last physical row is copied into the upper halo;
- the first physical row is copied into the lower halo;
- the last physical column is copied into the left halo;
- the first physical column is copied into the right halo.

Moving boundary handling outside the hot loop is a manual, algorithm-level
form of *loop unswitching*: the `periodic` condition is not evaluated for each
cell.

Injection also updates the corresponding halos when a source lies on a
periodic boundary, making newly injected energy visible across that boundary
during the next update.

The initial version additionally performed an injection before the loop when
the frequency was greater than one, but did not add it to `injected_heat`.
This unaccounted injection was removed: all injections now happen and are
recorded at the same point inside the loop.

Two modes support reproducibility:

- pseudorandom sources with a seed selected through `-s`;
- deterministic sources at one-quarter and three-quarters of the domain,
  selected through `-F`.

Benchmark scripts use the default pseudorandom mode and do not pass `-F`.
Deterministic sources remain available only for correctness tests.

## 7. Command-line parsing and validation

To keep parsing simple and close to the original template, arguments are
converted using `atoi`, `atol`, and `atof`. The program still checks for
positive domain sizes and iteration counts, a non-negative source count, and
non-negative energy.

As in the template, frequencies greater than one are capped at one. `-f 0`
means injection at every iteration. This favors simplicity but cannot
distinguish a non-numeric string from a genuine zero, nor diagnose overflow or
trailing characters. These limitations are accepted for this exercise.

## 8. Header cleanup

The original header contained many unnecessary includes, MPI constants unused
by the serial version, and public `initialize` and `memory_release` prototypes
that no longer matched the private functions implemented in the final source.

The final header introduces:

- the `STENCIL_SERIAL_FINAL_H` include guard;
- only the directly required `<stddef.h>` include for `size_t`;
- `static inline` linkage for functions defined in the header;
- removal of obsolete declarations and unused constants;
- conditional protection of OpenMP directives.

`static inline` avoids C inline-linkage issues when the same header is included
by different translation units. Each unit receives an internal definition that
the compiler may inline; actual inlining remains an optimizer decision rather
than a requirement imposed by the keyword.

## 9. Corrected grid dump

The template traversed the buffer as if halos did not exist: it used `size[0]`
as the stride and started at the first allocated cell. Its output therefore did
not represent the physical grid.

The final implementation:

- uses the actual `xsize+2` stride;
- starts at `(1,1)` and writes physical cells only;
- converts each row from `double` to `float` in a temporary buffer;
- opens the file in binary `wb` mode;
- handles the main allocation and file-opening failures.

The output remains a row-major sequence of `xsize*ysize` `float` values,
without halos.

## 10. Timing and performance metrics

A monotonic timer based on `clock_gettime(CLOCK_MONOTONIC)` measures separately:

- the `update_plane` kernel time;
- injection time;
- energy-reduction time;
- overall wall time.

Separating kernel time from the rest prevents I/O and diagnostics from
distorting the central measurement. Given:

```text
updates = xsize * ysize * niterations
```

the metrics are:

```text
GLUP/s  = updates / t_update / 10^9
GFLOP/s = 6 * updates / t_update / 10^9
```

The report also estimates bandwidth using a 24-byte-per-update model. This is
not the naive sum of five loads and one store: it assumes reuse through the
cache hierarchy and a particular model of memory traffic. It is therefore an
analytical model, not a measurement from hardware counters.

A CSV line enables automatic result collection without parsing the descriptive
output.

## 13. MPI extension and domain decomposition

The MPI version is implemented in `src/stencil_parallel_final.c` and reuses the
same optimized local kernel as the OpenMP version. The fundamental difference
is that every rank stores only one rectangular block of the global domain,
together with its halos. The full grid is not replicated on every process.

### 13.1 Two-dimensional process grid

Ranks are logically arranged in a `Px * Py` grid. Because the project allows
only a limited pool of MPI primitives, neither `MPI_Cart_create` nor
`MPI_Cart_shift` is used. The code selects the factorization manually by
testing every divisor of the process count and minimizing the local block
aspect ratio:

```text
local_x = global_x / Px
local_y = global_y / Py
score   = max(local_x/local_y, local_y/local_x)
```

A near-square local shape reduces the perimeter-to-area ratio: perimeter
determines communication volume, while area determines useful work. Linear
ranks are mapped to logical coordinates in row-major order:

```text
coord_x = rank % Px
coord_y = rank / Px
```

For example, 32 ranks over a square domain produce a `4 * 8` decomposition,
while 64 ranks produce `8 * 8`.

### 13.2 Non-divisible domains

Global dimensions need not be divisible by the process grid. For a global
dimension `N`, `P` blocks, and rank coordinate `c`, local size and offset are:

```text
base      = N / P
remainder = N % P
size(c)   = base + (c < remainder)
offset(c) = c*base + min(c, remainder)
```

The first `remainder` blocks receive one additional cell. The maximum imbalance
is therefore limited to one row or column. This behavior was also verified on
the non-divisible `101 * 97` smoke-test domain.

### 13.3 Manual neighbor discovery

Each rank directly computes its north, south, east, and west neighbors. With
non-periodic boundaries, an external neighbor is represented by
`MPI_PROC_NULL`; the corresponding communication is skipped and the halo stays
zero. With periodic boundaries, the neighbor beyond an edge is the rank on the
opposite side.

When a process-grid dimension is one, the code does not communicate with
itself. The kernel updates periodic halos for that direction locally. Corner
cells need not be exchanged because the five-point stencil uses only the four
orthogonal neighbors.

## 14. Halo exchange

At the beginning of each iteration, the current plane contains the state
produced by the previous iteration. Before computation, nearest-neighbor
communication updates all four halos.

North and south rows are already contiguous in row-major storage and can be
sent directly from the plane. East and west columns are non-contiguous. To
avoid `MPI_Type_vector`, which is outside the selected primitive pool, they are
explicitly copied into contiguous send and receive buffers. OpenMP
parallelizes column packing and unpacking when multiple threads are available.

The exchange sequence is:

1. pack the physical east and west columns;
2. post receives with `MPI_Irecv`;
3. post sends with `MPI_Isend`;
4. complete all requests with one `MPI_Waitall`;
5. unpack the received columns into the halos.

Distinct tags identify message directions and prevent ambiguous matches.
Posting non-blocking receives first makes matching buffers available early and
avoids dependencies on MPI buffering or rank execution order.

The current implementation does not overlap communication and computation:
`MPI_Waitall` completes before `update_plane`. Non-blocking primitives are used
to make the four-direction exchange safe and symmetric, not to hide latency.
A possible extension would compute the interior while messages are in flight
and update the four boundary strips afterward, but this would complicate the
kernel and is not required for the exam objectives.

## 15. Sources, reductions, and global correctness

Random sources are generated only by rank zero using the requested seed. Their
global coordinates are distributed with `MPI_Bcast`; each rank then keeps the
sources that fall inside its interval and converts them to local coordinates,
including the halo offset. Neither `Scatterv` nor rank-specific data structures
are needed.

Local energy is computed with the OpenMP reduction described in Section 5.
Local values are then summed on rank zero through:

```c
MPI_Reduce(..., MPI_SUM, 0, MPI_COMM_WORLD);
```

Allocation failures must also be handled collectively: if one rank exited
while the others continued, they could block at the next communication. The
`global_max_int` helper implements a logical OR through `MPI_Reduce` with
`MPI_MAX`, followed by `MPI_Bcast` to propagate the outcome. This intentionally
replaces `MPI_Allreduce`.

The smoke test ran periodic and non-periodic `101 * 97` domains with 1, 2, 4,
and 8 ranks. After 80 injections:

- the global periodic energy was exactly 80 for every decomposition;
- the non-periodic value was 79.9051 for every decomposition. A value below 80
  is expected because zero-valued external halos act as a heat sink.

Independence from the rank count jointly verifies source ownership, block
offsets, halo exchange, and global reduction. Smoke-test timings are too short
to have performance significance and are used only for correctness.

## 16. Hybrid MPI+OpenMP model

The program initializes MPI with:

```c
MPI_Init_thread(..., MPI_THREAD_FUNNELED, &level_obtained);
```

`MPI_THREAD_FUNNELED` is sufficient because all MPI calls are made by the main
thread outside OpenMP parallel regions. OpenMP parallelizes local computation,
initialization, column packing and unpacking, and the energy sum. Requesting the
more expensive `MPI_THREAD_MULTIPLE` level is therefore unnecessary.

On the Booster, each hybrid configuration keeps the total core count fixed and
varies the ratio of MPI ranks to OpenMP threads. The one-node sweep includes:

```text
1*32, 2*16, 4*8, 8*4, 16*2, 32*1
```

On two nodes, the total rank count doubles while the per-node layout is kept:

```text
2*32, 4*16, 8*8, 16*4, 32*2, 64*1
```

The first value is the total rank count, and the second is the number of
threads per rank. Scripts set `OMP_PLACES=cores`, `OMP_PROC_BIND=spread`, and
`OMP_DYNAMIC=false`; Slurm assigns a core block to each rank. The runtime
cannot dynamically change the thread count, and each configuration uses 32 or
64 physical cores on one or two nodes, respectively.

## 17. MPI primitives used

The final implementation deliberately relies on a restricted set of
primitives:

- `MPI_Init_thread`, `MPI_Comm_rank`, `MPI_Comm_size`, and `MPI_Finalize` for
  the MPI lifecycle;
- `MPI_Barrier` and `MPI_Wtime` for initial synchronization and timing;
- `MPI_Bcast` for sources and failure propagation;
- `MPI_Reduce` with `MPI_SUM` or `MPI_MAX` for energy, failures, and timings;
- `MPI_Irecv`, `MPI_Isend`, and `MPI_Waitall` for halo exchange.

The code uses no Cartesian topology, advanced collective, derived datatype, or
one-sided communication. The restricted pool makes domain decomposition and
data movement explicit, which are central concepts of the exercise.

## 18. Distributed timing and benchmark methodology

An `MPI_Barrier` precedes the timed region, after which every rank uses
`MPI_Wtime`. At the end, five local timings are reduced with `MPI_MAX`:

- `t_update`: local kernel only;
- `t_comm`: halo packing, communication, and unpacking;
- `t_inject`: local injection;
- `t_energy`: local energy computation and global reduction;
- `t_wall`: the complete timed simulation.

The maximum is the correct global metric because execution completes at the
speed of the slowest rank. Maxima for different metrics can originate from
different ranks, and communication and update timers are reported separately;
they must not be added afterward as though they were exact partitions of the
same maximum wall time.

MPI GLUP/s are computed from global updates divided by the maximum `t_update`.
Scalability plots instead use `t_wall`, which includes communication,
synchronization, and reductions. Let `n` be the number of nodes and `T(n)` the
wall time. The final campaign uses:

```text
strong speedup:     S(n) = T(1) / T(n)       ideal: S(n) = n
strong efficiency:  E(n) = S(n) / n          ideal: E(n) = 1
weak efficiency:    E(n) = T(1) / T(n)       ideal: E(n) = 1
```

Preliminary intra-node sweeps vary threads or ranks. They study binding,
bandwidth saturation, and layout selection, but do not replace the final
node-scaling experiment, where the per-node configuration must remain fixed.

All jobs use exclusive nodes, physical cores, `-O3 -march=native -fopenmp`, a
30-minute limit, and pseudorandom sources with the default zero seed. Each
result directory preserves its CSV, raw output, Slurm information, `lscpu`,
software environment, Git commit, and local patch.

The campaign includes:

- comparison of the template and final serial code with `-O1` and `-O3`;
- a reduced-grid serial test to expose cache effects;
- OpenMP strong scaling on a fixed `25000 * 25000` grid with `close` and
  `spread` binding;
- OpenMP weak scaling with 25 million cells per thread;
- preliminary MPI sweeps varying ranks on one and two nodes;
- rank/thread ratio sweeps on one and two nodes;
- final scaling from 1 to 16 nodes with four ranks per node and eight threads
  per rank.

The final strong-scaling test always uses `25000 * 25000` cells and 100
iterations. For weak scaling, the side of the square grid is:

```text
Nside(n) = round(25000 * sqrt(n / 16))
```

Since area is proportional to the square of the side, this formula keeps the
number of cells per node constant and reaches a maximum `25000 * 25000` grid
on 16 nodes.

## 19. Serial results

### 19.1 Large grid

The main comparison uses `25000 * 25000` cells, 200 iterations, and one
repetition per configuration. Data come from jobs
`49268411_template_serial` and `49293861_final_serial`:

| Variant | `-O1` [s] | `-O3` [s] | Speedup `-O1`/`-O3` |
|:--------|----------:|----------:|---------------------:|
| Template | 196.572 | 142.185 | 1.38 |
| Final | 207.229 | 153.376 | 1.35 |

Compiler optimization clearly reduces execution time for both versions. In
this single large-grid measurement, however, the final version is not faster
than the template: its `-O3` wall time is approximately 8% higher. This result
must be reported rather than hidden. The runs were executed in separate jobs,
so they cannot fully distinguish node-to-node variability from a genuine
regression. More importantly, with a working set much larger than the caches,
the stencil is dominated by memory traffic, and kernel simplification cannot
increase the available memory bandwidth.

The one-thread OpenMP `-O3` build takes 152.164 s, essentially the same as the
final serial `-O3` build. Enabling the OpenMP runtime therefore introduces no
significant one-thread overhead relative to the observed inter-job variation.

In all these measurements, `update_plane` accounts for more than 99% of wall
time: 141.384 of 142.185 s for the `-O3` template and 152.574 of 153.376 s for
the final `-O3` version. Performance analysis can therefore focus on the
stencil kernel.

### 19.2 Reduced grid and cache effects

The `1000 * 1000` test runs 10,000 iterations and contains three repetitions
per configuration. Together, the two planes occupy approximately 16 MiB,
excluding small metadata, and benefit much more from the shared cache than the
large grid.

| Variant | Build | Wall-time range [s] |
|:--------|:------|--------------------:|
| Template | `-O1` | 30.301–30.334 |
| Template | `-O3` | 11.106–11.170 |
| Final | `-O1` | 15.567–15.698 |
| Final | `-O3` | 6.582–6.896 |
| Final OpenMP, 1 thread | `-O3` | 7.125–7.200 |

Every repetition of the final `-O3` version is clearly faster than the `-O3`
template, with an improvement of approximately `1.6x`. Loop and memory-access
transformations become visible when the working set puts less pressure on
DRAM. The contrast between large and reduced grids is consistent with the
memory-bound nature of the stencil: an optimization can improve in-cache
execution without providing the same benefit after main-memory bandwidth is
saturated.

## 20. OpenMP results

### 20.1 Strong scaling and binding

Both binding policies start at approximately 152.16 s with one thread.
`spread` is clearly preferable up to 16 threads:

| Threads | `close` [s] | `spread` [s] | `spread` speedup |
|--------:|------------:|-------------:|-----------------:|
| 1  | 152.144 | 152.164 | 1.00 |
| 2  | 84.466  | 77.634  | 1.96 |
| 4  | 54.171  | 43.310  | 3.51 |
| 8  | 40.068  | 27.057  | 5.62 |
| 16 | 32.882  | 19.198  | 7.93 |
| 24 | 21.597  | 21.787  | 6.98 |
| 32 | 16.067  | 16.095  | 9.45 |

The advantage of `spread` through 16 threads is consistent with the CPU's SNC
layout: distributing threads engages both NUMA domains earlier and accesses a
larger fraction of the available bandwidth. The non-monotonic 24-thread point
shows that binding, first touch, and memory contention can matter more than the
core count alone. At 32 threads, speedup is `9.45x`, far below the ideal `32x`,
because all cores share the same DRAM bandwidth. Since every point contains
only one repetition, small differences should not be overinterpreted.

### 20.2 Weak scaling

The domain grows from `5000 * 5000` with one thread to `40000 * 20000` with 32
threads, keeping 25 million cells per thread. Wall time rises from 5.753 to
20.908 s, giving an overall weak efficiency of approximately 27.5%.

This does not indicate loop imbalance: every thread receives the same number
of cells. Instead, threads on one node share DRAM bandwidth. Increasing compute
resources does not proportionally increase available memory bandwidth, and the
stencil reaches its memory-bound regime. This experiment measures scaling over
threads inside one node, not weak scaling across independent nodes.

not the final node-scaling experiment: the first six points increase the core
count inside one node, and only the final point introduces a second node. It is
useful for showing memory saturation and halo-exchange cost, but it should not
be plotted against a node-based ideal line.

The homogeneous full-node comparison is 32 versus 64 ranks. Doubling nodes,
cores, and ranks reduces time from 16.168 to 8.056 s, a `2.007x` speedup.
Moreover, MPI and OpenMP produce almost identical 32-core times: 16.168 and
16.095 s. Intra-node halo packing and exchange therefore do not dominate the
kernel at this global size.

### 21.2 Weak scaling over ranks

Every rank owns 25 million cells. Wall time rises from 5.710 s with one rank to
approximately 21 s with 32 ranks on one node because all ranks share DRAM
bandwidth. The full-node comparison is instead nearly ideal:

```text
32 ranks, 1 node: 20.969 s
64 ranks, 2 nodes: 21.007 s
```

Weak efficiency from one to two full nodes is
`20.969 / 21.007 = 99.8%`. Every additional node brings an independent memory
subsystem, anticipating the behavior later confirmed up to 16 nodes.

## 22. Selecting the hybrid MPI+OpenMP layout

### 22.1 Fixed-problem sweep

On the `25000 * 25000` grid, all 32-core configurations take approximately
15.9 s, while all 64-core configurations take approximately 8.0 s:

| Ranks per node × threads per rank | 1 node [s] | 2 nodes [s] |
|----------------------------------:|-----------:|------------:|
| 1 × 32 | 15.957 | 8.035 |
| 2 × 16 | 15.965 | 8.006 |
| 4 × 8  | 15.901 | 8.010 |
| 8 × 4  | 15.891 | 8.019 |
| 16 × 2 | 15.898 | 7.969 |
| 32 × 1 | 15.921 | 7.963 |

Layout variation remains below one percent. At this scale, fewer messages from
using more threads are offset by OpenMP runtime cost and the packing of longer
columns. Pure MPI is marginally fastest on two nodes, while `8 × 4` is fastest
on one, but the differences are too small and based on only one repetition to
establish a meaningful ranking. The main result is robustness: all MPI/OpenMP
ratios exploit the available cores similarly.

### 22.2 Hybrid weak scaling on one and two nodes

In the hybrid weak-scaling test, the domain doubles together with the node and
core counts. Every per-node layout is compared with its two-node equivalent.
Wall times always remain between 15.83 and 15.98 s.

| Ranks per node × threads per rank | 1 → 2 node efficiency |
|----------------------------------:|----------------------:|
| 1 × 32 | 99.62% |
| 2 × 16 | 99.65% |
| 4 × 8  | 99.88% |
| 8 × 4  | 99.40% |
| 16 × 2 | 99.77% |
| 32 × 1 | 99.68% |

All configurations achieve efficiency close to 100%. The `4 × 8` layout has
the best value in this test and is a balanced compromise between message count
and OpenMP parallelism. It was therefore selected as the fixed configuration
for the 1–16 node campaign. This does not imply intrinsic superiority: the
sweep mainly shows that performance is not very sensitive to the MPI/OpenMP
mix for the measured problem.

## 23. Final hybrid MPI+OpenMP scaling across nodes

The final campaign is stored in
`results/49349184_mpi_node_scaling/go_mpi_node_scaling.csv`. Every node uses
four MPI ranks, eight OpenMP threads per rank, and all 32 physical cores. The
experiments use 1, 2, 4, 8, and 16 nodes, for a total of 4, 8, 16, 32, and 64
ranks. Each configuration contains one repetition.

### 23.1 Strong scaling across nodes

The problem remains fixed at `25000 * 25000` cells and 100 iterations:

| Nodes | Ranks | Process grid | `t_wall` [s] | Speedup | Efficiency |
|------:|------:|:-------------|-------------:|--------:|-----------:|
| 1  | 4  | 2 × 2 | 7.9589 | 1.000 | 100.0% |
| 2  | 8  | 2 × 4 | 4.0087 | 1.985 | 99.3% |
| 4  | 16 | 4 × 4 | 2.0350 | 3.911 | 97.8% |
| 8  | 32 | 4 × 8 | 1.1083 | 7.181 | 89.8% |
| 16 | 64 | 8 × 8 | 0.5185 | 15.350 | 95.9% |

Wall time falls from 7.959 to 0.518 s. The 16-node speedup is `15.35x` against
an ideal `16x`, for 95.9% efficiency. The curve stays very close to ideal over
the full range. The 8-node point loses more efficiency before recovering at 16
nodes. With one repetition, this non-monotonicity should not be treated as a
deterministic property of the code; node placement, network topology, and
system noise may contribute.

`t_update` alone decreases from 7.882 to 0.480 s, while wall time also includes
halo exchange and reductions. The difference between kernel and wall-time
scaling is the parallel overhead that prevents an exactly ideal curve.
`t_comm` remains between approximately 0.055 and 0.163 s and is not monotonic.
These values are maxima that may come from different ranks and, again, from a
single measurement.

### 23.2 Weak scaling across nodes

The grid grows with the square root of the node count. Rounding its side causes
only a negligible difference in workload per node:

| Nodes | Global grid | Cells per node | `t_wall` [s] | `t_update` [s] | Efficiency |
|------:|:------------|---------------:|-------------:|---------------:|-----------:|
| 1  | 6250 × 6250   | 39,062,500 | 0.4894 | 0.4800 | 100.0% |
| 2  | 8839 × 8839   | 39,063,960 | 0.5163 | 0.4805 | 94.8% |
| 4  | 12500 × 12500 | 39,062,500 | 0.5123 | 0.4798 | 95.5% |
| 8  | 17678 × 17678 | 39,063,960 | 0.5610 | 0.4707 | 87.2% |
| 16 | 25000 × 25000 | 39,062,500 | 0.5267 | 0.4881 | 92.9% |

Local kernel time stays within 0.471–0.488 s, confirming a constant and
well-balanced computational workload per node. The larger wall-time variation
therefore comes from communication, synchronization, and reductions rather
than increased local work. Weak scaling also reaches its minimum efficiency at
8 nodes and recovers at 16; the final 92.9% shows that multinode overhead
remains limited.

In all ten runs, injected and final energies match at 100, as expected for 100
iterations, one source, and the particular non-periodic case in which the
random source does not lose a measurable amount of energy through the boundary
over the simulated time. This agreement is a useful check, although the
periodic/non-periodic smoke test in Section 15 remains the stronger validation
of boundary correctness.

## 24. Makefile, Slurm jobs, plots, and reproducibility

The target:

```text
make mpi
```

compiles `src/stencil_parallel_final.c` with `mpicc`, using the same
`-O3 -Wall -Wextra -march=native -fopenmp -Iinclude -g` flags as the OpenMP
version. Slurm scripts are collected under `sbatch/` and must be submitted from
the repository root, for example with
`sbatch sbatch/go_mpi_node_scaling.sh`. The dedicated scripts are:

- `sbatch/go_mpi_smoke.sh`: correctness with multiple ranks, periodic
  boundaries, and a non-divisible domain;
- `sbatch/go_mpi_strong.sh`: preliminary MPI sweep from 1 to 64 ranks;
- `sbatch/go_mpi_weak.sh`: preliminary weak scaling with 25 million cells per
  rank;
- `sbatch/go_mpi_hybrid.sh`: fixed-problem rank/thread ratio sweep;
- `sbatch/go_mpi_hybrid_weak.sh`: paired hybrid weak scaling on one and two
  nodes;
- `sbatch/go_mpi_node_scaling.sh`: final strong and weak scaling from 1 to 16
  nodes with a fixed `4 × 8` configuration per node.

Each script produces one CSV containing run name, resource configuration,
maximum timings, GLUP/s, and injected and measured energies. Raw output is
preserved to verify both parsing and energy conservation. Environment setup
uses `module purge` before loading OpenMPI, avoiding the conflict between
different GCC modules observed during the first smoke test.

`plots.py` recursively searches for CSV files under `results/` and creates
English-language figures with ideal reference lines. For the final campaign it
produces, in particular:

- `plots/12_mpi_nodes_strong_scaling.png`, node-based speedup;
- `plots/13_mpi_nodes_weak_efficiency.png`, node-based weak efficiency.

Result directories are identified by their Slurm job IDs. The final CSV copied
into the local repository contains only measured data and does not reconstruct
missing fields.

## 25. Limitations of the experimental analysis

The results are strong, but they must be presented together with their limits:

1. almost every point contains one repetition, so no error bars or robust
   variability estimate are available;
2. software timers measure program sections but do not replace hardware
   counters for DRAM bandwidth, cache misses, or vectorization;
3. communication and computation are not overlapped: `MPI_Waitall` precedes
   the kernel, so overlap optimization was not evaluated;
4. the campaign uses one node type, one compiler, and one binding scheme;
   quantitative conclusions should not be transferred automatically to other
   architectures;
5. `4 × 8` was selected as a balanced layout, not as a statistically proven
   winner of the sweep;
6. the small non-monotonic behavior at 8 and 16 nodes is compatible with noise
   and placement effects, but repeated measurements would be required for a
   confident attribution.

## 26. Conclusions

The final implementation regularizes kernel memory accesses, exposes
non-aliasing, uses aligned allocations, and introduces OpenMP and MPI
parallelism without changing the numerical operator. On the reduced domain,
where the working set benefits from cache, the final serial `-O3` version is
approximately `1.6x` faster than the template. This advantage does not appear
in the single available large-domain measurement: `update_plane` accounts for
more than 99% of execution time, and main-memory bandwidth dominates behavior.

Within one node, increasing threads or ranks quickly saturates shared memory
bandwidth. Pure OpenMP and MPI achieve almost identical times on all 32 cores,
and the hybrid sweep shows less than one-percent variation among rank/thread
ratios. Layout selection therefore has a secondary effect compared with the
number of cores and, especially, the number of available memory subsystems.

Across nodes, every additional resource provides new cores and new DRAM
bandwidth. With a fixed layout of four ranks and eight threads per node, strong
scaling reaches `15.35x` on 16 nodes, or 95.9% efficiency. In weak scaling, the
local kernel time remains nearly constant and 16-node efficiency is 92.9%.
These results demonstrate near-ideal multinode scalability for the studied
problem size while still exposing communication overhead and single-run
variability.

Finally, the distributed implementation retains a restricted set of MPI
primitives: decomposition and neighbors are computed manually, columns are
packed without derived datatypes, and global reductions are limited to
correctness and timing. The result meets the educational objective of the
exam: connecting kernel optimization, memory hierarchy, binding, domain
decomposition, and correct strong- and weak-scaling measurements.
